# `rr_parity` — bngsim vs RoadRunner (SBML, ODE + SSA)

Validates bngsim's SBML engine against [libRoadRunner] on the BioModels
keep-set. The SBML loader is on trial alongside the engine: a parser divergence
surfaces as `EXCEPTION`/`DIFF`, not a silent pass.

## Layout

```
rr_parity/
  models/<model_id>/           one dir per model
    *.xml                      BioModels SBML  — GITIGNORED, ~270 MB (materialize.py)
    *.sedml                    curated SED-ML  — GITIGNORED, ~49 MB (temp-biomodels, PMC12677764)
  build_ode_jobs.py            emits ode_jobs.json (1323 jobs) — the ODE SPEC
  build_ssa_jobs.py            emits ssa_jobs.json — the SSA golden SPEC (rr_golden only)
  materialize.py               places SBML+SED-ML into models/ from local sources
  ode_jobs.json                _core ODE manifest (model × ode × roadrunner)
  ssa_jobs.json                _core SSA manifest — consumed by rr_golden only (GH #108)
  _rr_common.py                engine adapters + column alignment + isolation scheduler
  rr_run.py                    ODE runner: both engines → _core report (the RESULTS)
  rr_golden.py                 golden emitter: bngsim-only checksum+fingerprint (+traj subset)
  golden/golden.json           golden references — regenerated by consumers (not committed)
  runs/                        per-run _core reports — GITIGNORED
  ssa_screen.py                broad SSA cross-engine screen (GH #30/#107/#109), attribution-rich
  build_ssa_candidates.py      derives the biomodels SSA corpus + structural gates
  ssa_candidates.json          the biomodels SSA corpus (306 models; ID+metadata only) — per-model SPEC
  ssa_spec.json                canonical global run config — the run-config SPEC
  ssa_attribution.py           maps the screen's statuses onto _core Outcome + subclass
  make_ssa_baseline.py         promotes a screen run to ssa_baseline.json
  ssa_baseline.json            committed _core baseline (expected per-model outcomes) — the BASELINE
  runs/ssa_report.json         a fresh screen run as a _core report — GITIGNORED
```

### SSA screen → `_core` taxonomy (`ssa_attribution.py`)

The screen's status vocabulary is richer than `_core.Outcome`, so
`ssa_attribution` is the single bridge that re-expresses each run as a `_core`
report (`ssa_screen.py` writes both: the native `ssa_screen.json` keeps the
fuller coverage detail, `ssa_report.json` is the shared-schema view). The mapping
is **honest** — an oracle-attributed RR-side divergence stays `DIFF`, never
collapses into `PASS` — and records the attribution in `JobResult.subclass`:

| screen status | `_core` Outcome | subclass | scored |
|---|---|---|---|
| `pass` | `PASS` | — | green |
| `fail` | `DIFF` | `bngsim_suspect` | **red** |
| `diff_ode_level` | `DIFF` | `ode_level` | **red** |
| `diff_not_bngsim` | `DIFF` | `diff_not_bngsim` | green |
| `rr_known` | `DIFF` | `rr_known` | green |
| `too_slow` | `TIMEOUT` | — | green (coverage gap) |
| `error` | `EXCEPTION` | — | **red** |
| `skip` | `SKIP` | — | green |

A suite scoring policy (`is_expected(outcome, subclass)`) decides green/red — it
greens the attributed-not-bngsim DIFFs, the `too_slow` coverage gap (deliberately
overriding `_core`'s generic `TIMEOUT ∈ FAILING`), and a RoadRunner-side refusal
(`error` whose message is `roadrunner:`-prefixed → `REFERENCE_FAILED`/`feature_gap`,
since bngsim ran where RR couldn't), and reds exactly the screen's own exit-code
set (a `bngsim_suspect` / `ode_level` DIFF, a bngsim-side EXCEPTION). The GH #108
regression diff is built on `is_regression(...)` (a model crossing from an
expected state into a scoring DIFF/EXCEPTION).

### SSA timing (GH #135)

Each screen case also records a **measurement-only** per-engine timing dict
(`JobResult.timing` = `{warmup, bngsim, roadrunner}`), rendered by
`generate_matrix.py --regime ssa` into an HTML matrix that **deliberately mirrors
the ODE matrix** (`generate_html`) — same BNGsim-vs-RoadRunner three-column
layout, green/yellow/red/gray color convention, model cell
(species/reaction/parameter/compartment counts + feature tags), top-of-page cost
plots + wins tables + summary boxes, and bottom-of-page legend — so a reader
moving between the two is never disoriented. Only the regime-specific content
differs: the agreement metric is a mean z-score (not a relative trajectory
difference), and the timing taxonomy is SSA's:

- **① Per-process startup** — one-time library import per worker. bngsim's SSA
  warmup is the `_bngsim_core` extension load (the SSA path derives **no**
  Jacobian and imports **no** SymPy, unlike ODE); RoadRunner's is its LLVM/JIT
  engine init (gillespie still JITs). Captured by `measure_ssa_warmup()` before
  any engine call.
- **② Per-model load** — for bngsim, **parse + interpret only** (no Jacobian, no
  codegen), read from the public `Model.last_libsbml_parse_sec` /
  `last_interpret_sec` accessors; for RoadRunner, `getLoadTimings()`
  (read/parse/interpret/codegen/JIT), JIT-dominated.
- **③ Per-replicate simulation** — the ensemble runs N independent reseeded
  trajectories (model loaded once, cloned/reset per replicate), so there is **no**
  cold/warm-reuse split of a single solve like the ODE matrix. The renderer
  reports the replicate-time distribution plus the rep-1 **cold** vs warm
  (reps 2..N) median (`_ensemble_stats`). bngsim's per-replicate clone+Simulator
  setup is shown separately (`setup_per_rep_sec`) so it never contaminates the
  per-trajectory comparison; RoadRunner reuses its one compiled model.

**Ordering axis — stochastic activity.** The matrix sorts the rows and lays out
the plot x-axis by **reaction events per unit simulated time** (BNGsim's
`solver_stats["n_steps"]` summed over the ensemble ÷ N·t_end), not the species
count — that activity is what drives per-trajectory cost (a 1-species model that
fires 10⁴ events/time costs far more than a 50-species near-static one). Each
model cell shows its rate; `events_per_time` lives in the BNGsim timing dict.

The most active models are exactly the ones that **time out** (TOO SLOW), and a
killed worker reports no ensemble timing — so they would otherwise show a blank
activity. `_activity_probe` fills that in: before the (possibly long) ensemble,
each worker estimates the firing rate from a short window (simulate from t=0 under
a tight wall cap; if the engine raises `SimulationTimeout`, shrink the horizon and
retry — a hyper-active model fires enough events in a tiny window to finish fast)
and writes it to a gitignored sidecar (`runs/_probe/<id>.json`) that survives the
kill; `_schedule` recovers it on the `too_slow` branch. Those rows show the rate
flagged as a **bounded-window probe** estimate and sort as high-activity. The
probe runs separately-seeded clones, so it never perturbs the ensemble's
trajectories or z-scores; it adds a small per-model wall overhead that can nudge a
borderline model across the timeout cap (non-scoring churn, never a regression).

**Timeout adjudication — every model gets the best verdict it can.** A raw
"too slow" conflates three very different things, so the worker bounds each
replicate (`REP_WALL_CAP`, the per-replicate wall budget ≈ 40 s / (N×2 engines))
and resolves each model to the strongest verdict possible:

- **Full verdict** — every replicate finishes the full horizon within the cap →
  the normal PASS / DIFF (with ODE attribution) — green / yellow / red.
- **Recovered verdict** — the SSA finishes but a *fail*'s deterministic-ODE
  attribution hangs on a stiff oracle (e.g. MODEL1909300003, whose SSA fires 62
  events but whose ODE never converges). The worker persists the completed
  verdict to a sidecar *before* attribution (a sidecar, not a queue put, because a
  feeder thread cannot flush while the C++ solve holds the GIL); on a kill,
  `_schedule` recovers it as `attribution_incomplete` — yellow: we found a
  divergence but could not assign fault.
- **Partial-horizon verdict** — a replicate never finishes the full horizon (a
  uniformly hyper-active model, OR a *seed-dependent* runtime sign-indefinite rate
  explosion — MODEL1909300003's seed 2007 explodes while every other seed fires
  62 events). `_partial_adjudicate` shrinks the horizon ×1/10 past each blowup to
  the largest `[0,H]` at which **all** replicates of both engines finish, and
  compares there → `partial_horizon` (yellow), labeled `t=0→H`. bngsim is bounded
  and runs first; RoadRunner only once bngsim succeeds at `H` (so it is fast too).
- **Coverage gap** — even a 1 %-of-horizon window has a replicate that never
  finishes → a genuine gray `TIMEOUT`, with its measured activity shown.

This replaced the old all-or-nothing `too_slow`: 109 gray timeouts became 63 gray
(genuinely intractable) + 52 yellow partial-horizon verdicts + recovered full
verdicts. `partial_horizon` / `attribution_incomplete` are scored **triaged**
(provisional, never a red bngsim defect), so they never regress; `ssa_baseline.json`
was re-based to the adjudicated outcomes.

The instrumentation never touches a verdict or metric — the regression diff
(`diff_ssa_baseline.py`) is byte-identical with or without it. RoadRunner is one
fixed config point (LLVM-JIT · gillespie), shown as a single reference column.

### Spec / baseline / report (the regression's three files)

Mirroring the suite's manifest-vs-report split, the SSA screen has three
committed-or-not artifacts:

- **Spec** — *which* models (`ssa_candidates.json`, the per-model corpus) and
  *how* to run (`ssa_spec.json`, the canonical global config: corpus/effort/n/
  seed/horizon/tolerances/wall cap). Both committed and stable.
- **Baseline** — `ssa_baseline.json`, the committed `_core` report of the
  *expected* per-model outcomes at the spec config. Re-baseline deliberately:
  `python make_ssa_baseline.py --from runs/ssa_screen.json` after a reviewed and
  accepted change in outcomes. The seed was the canonical legacy result
  (`harness/results/rr_ssa_biomodels_parity.json`, now retired): 318 models →
  197 PASS, 7 green DIFF (5 `diff_not_bngsim` + 2 `rr_known`), 109 TIMEOUT, 5
  REFERENCE_FAILED, **0 not-expected**.
- **Report** — `runs/ssa_report.json`, a *fresh* run's `_core` report
  (gitignored). The regression diff (`diff_ssa_baseline.py`) compares this against
  the baseline; it exits non-zero only on a model crossing into a scoring
  DIFF/EXCEPTION.

The regression is gated in CI by `python/tests/test_ssa_baseline_regression.py`:
a **hermetic** layer (the committed baseline is well-formed + all-green, is
self-consistent under the diff, the diff demonstrably catches a synthetic
regression, and the baseline matches `ssa_spec.json`) that always runs, plus an
**opt-in** 2-model engine slice smoke that `skipif`s when RoadRunner or the
biomodels SBML corpus is absent. The full 318-model breadth run stays a manual job
(`ssa_screen.py` at the `ssa_spec.json` config).

## Why both SBML and SED-ML are gitignored

The SBML is the full BioModels keep-set — ~270 MB that would bloat git history
forever. The SED-ML are the curated simulation protocols from the temp-biomodels
paper (PMC12677764); they carry the per-model time grids / tolerances the SBML
itself lacks. Both are re-fetchable from their sources, so both stay out of
version control and are re-placed locally — a working checkout is *expected* to
hold local copies of both:

```sh
python materialize.py                 # places SBML + SED-ML (both gitignored)
python materialize.py --sedml-only    # just refresh the SED-ML
```

Sources resolve from `$BIOMODELS_SEDML_DIR` (temp-biomodels mirror) and
`$BIOMODELS_SBML_DIR` (a local BioModels download). materialize.py only copies —
it never generates protocols — so the placed files are byte-identical to the
mirror. To version-control either set, drop the matching line in
`models/.gitignore`.

### Pinning & provenance (reproducibility)

The temp-biomodels fetch is **pinned to an exact commit** so "reproduce our
numbers" is deterministic. `fetch_sedml.py` checks out
`6a09daf46af1bb89e4857436b623a7b8720863ad` (sys-bio/temp-biomodels, CC0-1.0) by
default; `--upstream-latest` refreshes against `main` but warns about drift. The
committed `temp_biomodels_manifest.json` records every referenced file's pinned
commit + path + sha256 (1983 rows: 992 SED-ML + 991 SBML), and `fetch_sedml.py`
verifies the fetched bytes against it. Regenerate the manifest with
`generate_sedml_manifest.py` (schema: `parity_checks/manifest.schema.json`).

The 331 `invented` protocols are **authored by us** — no upstream file, so they
are not in this manifest; their grid/tolerances live inline in the committed
`ode_jobs.json`. The 332 `biomodels_dir` SBML come from BioModels REST and are
pinned by the biomodels suite's version manifest
(`../../benchmarks/suites/biomodels/biomodels_provenance.json`) — all 332 are
inside that keep-set, so `materialize.py`'s `$BIOMODELS_SBML_DIR` copy is
reproducible once that suite's pinned `fetch.py` has run.

## Comparison protocol (shared with bng_parity, in `_core.differ`)

The runner gates on `_core.differ` — the **same** verdict bng_parity tuned to
its 895/895 golden — not a bare worst-cell metric. Two pieces:

- **ODE — `deterministic_verdict`.** The solver error model `|a−b| ≤ atol +
  rtol·|y|` as a combined per-cell tolerance (`REL_TOL=1e-4` relative +
  column-peak and file-scale absolute terms), then a **fail-fraction budget**
  (`FAIL_FRAC_BUDGET=5e-3`: a few stiff-transient cells may exceed the soft bar)
  gated by **hard ceilings** (`HARD_REL_CEILING=0.05` on magnitude-carrying
  cells, `HARD_ABS_CEILING_FILE=1e-2·file_scale`) so a concentrated real
  divergence is never forgiven. Reported metric: `max_rel_err` = post-budget
  worst remaining relative error (0.0 on PASS).
- **SSA — handled by `ssa_screen.py`, not `rr_run`.** `rr_run.py` is ODE-only
  (GH #108); the cross-engine SSA verdict is the screen's z-gate + third-oracle
  attribution (see "SSA screen → `_core` taxonomy" above). `_core.differ`'s
  `ensemble_verdict` is still used by `rr_golden.py` (SSA golden) and bng_parity.

### Integration tolerance — forced tight, shared by both engines

Both engines run at a single **`rtol=1e-9, atol=1e-12`** (override `--rtol/--atol`).
This is the SBML analog of bng_parity's shared `atol/rtol=1e-8` default — but
tighter, because BioModels run in **concentration space** (many species at
`1e-5`–`1e-9`), where `1e-8` would swamp the signal. The per-job SED-ML
`rtol/atol` stay in the manifest as horizon/figure **provenance only** and are
not used for the parity run. Rationale: parity asks "do the engines compute the
same trajectory" — best answered by integrating both as accurately as possible.
At the SED-ML default `rtol=1e-6`, two independent stiff integrators legitimately
diverge from *each other* at ~1e-3 (each within its own tolerance); on a 30-model
sample, forcing the tight shared tol takes ODE from 14 DIFF (bare `max_rel_err`)
down to **1**, the lone residual being a genuine divergence (BIOMD0000000019,
Schoeberl EGFR — 1855 hard-ceiling cells). A model that genuinely can't integrate
at the tight tol surfaces as `EXCEPTION`; the bng_parity-style per-model
"tighten/loosen both stacks, with a reason" override is the escape hatch (TODO).

The ODE keep-set definition (`manifest.csv`) still lives under
`benchmarks/suites/biomodels/` (override via `$BIOMODELS_SUITE_DIR`). The SSA
corpus `ssa_candidates.json` is now vendored here beside `build_ssa_jobs.py` /
`ssa_screen.py` (override via `$SSA_CANDIDATES_JSON`). The screen's roundtrip
corpus (BNGL-derived SBMLs) and its `suite_ssa.json` manifest remain under
`benchmarks/` (override via `$SBML_ROUNDTRIP_DIR` / `$SUITE_SSA_JSON`).

## Instrumented RoadRunner — per-phase load timing (GH #135)

Unlike AMICI (which self-times every build phase via `@log_execution_time`),
stock RoadRunner fuses parse + interpret + LLVM-IR-gen + JIT into one opaque
`RoadRunner(xml)` call — there is no public API to split them. Getting an
honest per-phase load breakdown for the reference engine therefore needs a
**source-patched build**.

That build exists at our fork:

- **`github.com/wshlavacek/roadrunner`, branch `timing-instrumentation`**
  (off `develop`; commits `b11ca753d` + `e5332d90c`).
- `RoadRunner::getLoadTimings()` (`source/rrRoadRunner.h`) → a `dict[str,float]`
  of seconds for the **most recent** `load()`, repopulated each construction:
  `read_sec` (fetch SBML text) · `parse_sec` (libSBML parse) · `interpret_sec`
  (SBML doc → model symbols) · `codegen_sec` (LLVM IR gen) · `jit_sec` (JIT
  compile/materialize) · `model_cache_hit` (1.0 if served from the in-process
  model cache so codegen/JIT were skipped). The matrix's RoadRunner "codegen"
  analog = `codegen_sec + jit_sec`.
- `roadrunner.__warmup_sec__` (`wrappers/Python/roadrunner/__init__.py.in`) →
  the per-process `import roadrunner` cost (dlopen + static init), set once at
  import. `_rr_common.py` prefers it over the in-house import-proxy estimate.
- The patch is **measurement-only** — A/B simulation fingerprints are
  bit-identical to stock 2.9.2 (it only wraps `std::chrono` around unchanged
  calls), mirroring the bngsim no-hot-path gate.

**Graceful degradation — the sweep runs on stock RoadRunner too.** When the live
`roadrunner` lacks `getLoadTimings()`/`__warmup_sec__` (any unpatched 2.9.2,
e.g. the `pip`/`conda` wheel), `rr_ode`/`ssa_screen` fall back automatically:
the RoadRunner load collapses into a single lumped `parse_interpret_codegen_sec`
(rendered "—" for the sub-phases in the matrix), warmup falls back to the
import-proxy estimate (`warmup.roadrunner_source == "import-proxy"`), and the
per-phase RR `*_sec` keys read 0. **bngsim's full load split is unaffected.** So
the parity verdict and the bngsim timing are always complete; only the
*RoadRunner load decomposition* depends on the instrumented build.

**Installing it (optional).** The build is verified on both **Intel macOS
(x86_64)** and **arm64 (Apple silicon)**; `build_instrumented_roadrunner.sh`
(next to this README) automates the arm64 build end-to-end (override
`RR_BUILD_ROOT` / `BNGSIM_VENV`). The full reference — the 3-stage build recipe
(prebuilt LLVM-13 → `libroadrunner-deps` non-recursive submodules → roadrunner
with `BUILD_PYTHON=ON`), the venv-shadow install procedure, the caching/warmup
cost model, and the RoadRunner method-detection facts — lives in
[`INSTRUMENTED_ROADRUNNER.md`](INSTRUMENTED_ROADRUNNER.md). Once built, the
compiled `roadrunner/` package shadows the wheel in `.venv` and the next sweep
picks up the split with no code change.

## Running the sweep (`rr_run.py`)

Both engines are in-process Python, so the runner compares them directly — no
two-output-tree diff like bng_parity. Each job runs in its own spawned
subprocess that the parent kills on wall-clock overrun, so a RoadRunner
segfault on a pathological model can't take down the screen.

`rr_run.py` is **ODE-only** (GH #108 retired its `--regime ssa` branch; the SSA
cross-engine screen is `ssa_screen.py`).

```sh
cd bngsim
python materialize.py                               # place the gitignored model tree first
.venv/bin/python parity_checks/rr_parity/rr_run.py --limit 20 --workers 4
.venv/bin/python parity_checks/rr_parity/rr_run.py --models BIOMD0000000012,BIOMD0000000010
```

- `--limit`, `--models`, `--include`/`--exclude` (substring on the model path),
  `--workers`, `--timeout` (per-job wall cap; ODE jobs default to 120 s),
  `--rtol`/`--atol` (override the forced ODE integration tolerance).
- Writes a `_core` report to `runs/report_ode.json` (gitignored). Every
  result stamps bngsim + roadrunner + python/platform versions, and `_meta`
  records the forced `integration_tol`.
- **Outcome mapping** (one taxonomy with the other suites): protocol PASS →
  `PASS`; protocol fail / non-finite / disjoint species set / time-grid mismatch
  → `DIFF`; bngsim `SsaValidationError` (a clean `validate_for_ssa` refusal) →
  `UNSUPPORTED`; either engine raised or failed to load → `EXCEPTION`; wall-cap
  overrun → `TIMEOUT`. An SBML-loader divergence is always a real
  `EXCEPTION`/`DIFF`, never a silent pass.
- **REFERENCE_FAILED sub-class** (`JobResult.reference_refusal`, #94): the
  "bngsim ran, RoadRunner refused" bucket is auto-split by WHY RR refused, so a
  later run skips the decided cases and only re-triages the unverified ones.
  `overstrict_missing_value` is **settled** — RR refuses a valueless parameter
  that bngsim provably-correctly accepts (post-#94 the loader hard-rejects a
  *referenced* genuinely-missing param, so a model bngsim still ran had an
  unreferenced / rule-defined one). `feature_gap` / `integrator` / `recursive`
  are oracle-unverified and triage-worthy. `_meta.reference_refusal_breakdown`
  is the regression tracker — the settled count climbing back means the loader
  gate regressed.

### Toward zero DIFF

With the tight shared integration tolerance and the `_core.differ` protocol, the
solver-disagreement DIFFs that a bare `max_rel_err` flagged collapse to the small
set of **genuine** cross-engine divergences (e.g. BIOMD0000000019; sub-particle
SSA degeneracies where bngsim floors <1-molecule species to 0 while RR gillespie
keeps the fraction). Those are catalogued in `dev/notes/SBML_VS_ROADRUNNER.md`
(AssignmentRule-on-species, hOSU=true V≠1, RR-gillespie lag, sub-particle floor).
Reaching a clean headline count means either fixing them as loader/engine bugs or
moving the known/accepted ones to a `KNOWN_ARTIFACT`-style allow-list with a
reason each — exactly as bng_parity does. (Allow-list + per-model tolerance
overrides are the next addition; not yet wired.)

## Emitting golden references (`rr_golden.py`)

Pure bngsim output — no RoadRunner side, no differ. Consumers (PyBNF,
PyBioNetGen) regenerate these through their own bridges on a pinned bngsim to
verify they reproduce the engine, *without* needing a RoadRunner install.

```sh
.venv/bin/python parity_checks/rr_parity/rr_golden.py --regime both --workers 4
```

- One trajectory per job. `checksum` = sha256 of the trajectory rounded to 9 sig
  figs (byte-identity within a pinned version/platform/seed cell — verified
  reproducible across independent runs for ODE *and* SSA). `fingerprint` = the
  per-variable numeric summary (`fingerprint_max_rel`-comparable), the
  cross-platform tolerance fallback.
- Stochastic = a single fixed **`seed=1`** trajectory, NOT an ensemble.
- Full trajectories only for the hand-picked `TRAJECTORY_ALLOWLIST` (each with a
  one-line reason); every other job gets checksum + fingerprint. Trajectory
  files are downsampled if needed to stay under the 1 MB large-file hook;
  `golden.json` keeps full-resolution checksum + fingerprint regardless.
- `golden/` is regenerated by consumers (not committed; `rr_run.py` compares both
  engines live). `runs/` is gitignored.
